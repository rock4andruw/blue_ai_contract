# 合約智能比對助理 — 專案說明

**最後更新**: 2026-06-25  
**定位**: 讓法務與 PM 在 **30 分鐘內**完成過去需要 2 小時的合約審查  
**一句話**: 「像有一個資深法務顧問幫你審合約」  
**當前狀態**: 🟢 Phase 1 完整版完成（核心管道 + FastAPI + Demo UI + 三層協商對策）

---

## 一、問題有多大？（簡報數據）

### 時間浪費

> 專業法務人工審閱一份 NDA，平均耗時 **92 分鐘**；整體合約週期手動處理平均需 **3.4 週**。
>
> — LawGeex vs 20 Lawyers Study；WorldCC / Agiled CLM Statistics

### 影響範圍廣

> **87%** 法務人員認為 AI 最能幫助的是 pre-signature 審查與 redlining；
> **56%** 法務團隊處理一份標準 NDA 需要超過**一週**；
> **49%** 仍使用 email + Word + 共用資料夾管理合約版本。
>
> — LegalOn 2026（n=452）；SpotDraft 2025（n=115）

### 財務代價高

> 低效的合約流程平均延誤新產品上市 **6.5 天**，造成 **700 萬美元**營收損失；
> 合約管理不善使企業流失高達 **9%** 年營收。
>
> — Gartner Research；McKinsey & Company

### 台灣同樣面臨相同問題

> 工研院指出台灣法務人員面臨**知識累積不足、合約審閱負擔重**；
> 勤業眾信台灣調查顯示企業普遍有**合約數量暴增、缺乏管理系統、審閱吃緊**三大困境；
> 先行智庫訪查指出**版本控管缺失**為台灣企業五大痛點之一。
>
> — 工研院《企業合約文本 AI 智慧處理技術》；勤業眾信台灣；先行智庫

### 我們的解法

> **三段式提速**：AI 初審 26 秒完成差異比對與風險標記 → 人工確認 30 分鐘以內 → 附協商對策直接談判
> 不只找差異，還告訴你哪個重要、風險多高、怎麼談。
>
> — 對比基準：LawGeex 研究（人工 NDA 審閱 92 分鐘，AI 26 秒，AI 準確率 94% vs 人工 85%）

## 二、解決什麼問題

每次供應商送來新版合約，法務或 PM 需要：

1. 人工逐條比對新舊版本（1-2 小時）
2. 判斷哪些變更有風險（需要法律知識）
3. 想出協商對策（需要談判經驗）

**現有工具的延伸空間**：Lumine AI 已提供合約差異比對的核心能力，Blue-AI 以此為基礎，往上疊加三層價值：**重點收斂**（100 個差異 → 3-5 個主要變更）、**風險分級**（高／中／低，附觸發原因）、**協商對策**（每個高風險給出 2-3 個可直接用於談判的方案）。兩者定位互補——Lumine AI 負責文件層面的差異偵測，Blue-AI 負責業務層面的風險判讀與決策支援。

**台灣研究缺口**：國際上已有充分量化數據證明合約審查是高耗時、高 ROI 的 AI 導入場景；台灣本地已有工研院、勤業眾信等質性研究確認相同痛點，但**缺乏針對台灣企業 CLM 工時與審查週期的本地量化研究**。本系統以台灣常見的 SLA / NDA / 採購合約為對象，填補這個研究與實務的空白。

## 三、我們做了什麼

### 核心功能（已實作）

| 功能 | 說明 |
| --- | --- |
| **差異比對** | 逐條找出新增／修改／刪除條款（LCS + Needleman-Wunsch 對齊） |
| **智能摘要** | 100 個差異 → **3-5 個主要變更**，濾掉行政性小改動 |
| **風險分析** | 高／中／低三級標示，含商業影響說明，100% 高風險召回 |
| **三層協商對策** | 按需生成（點擊觸發）：🟢 首選立場 / 🟡 可接受妥協 / 🔴 絕對底線，附替換條款文字 |
| **歸因標籤** | 每個變更標示「乙方新增／修改／刪除條款」 |
| **原文對照** | 可展開查看原始條款 vs 修訂條款並列對比 |
| **HTML 追蹤修改清理** | 自動移除 Word 匯出的 `<del>/<ins>` 標記，回復乾淨文字 |
| **Demo UI** | 靜態單頁，上傳 / 範例模式（v2-v5），含 Hero 指標條 |
| **FastAPI** | `POST /api/v1/contracts/compare`、`POST /api/v1/contracts/negotiate`，JSON 回傳 |

### 與 Lumine AI 的功能定位

Blue-AI 以 Lumine AI 的差異比對能力為基礎，往上延伸三層決策支援：

| 功能層次 | Lumine AI | Blue-AI（延伸） |
| --- | --- | --- |
| 條款差異偵測 | ✅ 核心能力 | ✅ 沿用 |
| 重點摘要（3-5 點） | — | ✅ 新增 |
| 風險等級標示（高／中／低） | — | ✅ 新增 |
| **協商對策建議** | — | ✅ 新增 |
| **可量化召回率驗證** | — | ✅ 新增 |
| **Skill Sub-Agent 架構** | — | ✅ 新增 |
| 繁體中文 SLA / NDA 優化 | — | ✅ 新增 |

> Blue-AI 可作為 Lumine AI 的下游分析模組，未來可直接整合進 Lumine AI 工作流，使用者在 Lumine AI 取得差異清單後，一鍵進入 Blue-AI 進行風險研判與協商建議。

### 技術架構

```text
[Demo UI：frontend/demo.html]
  上傳模式（drag & drop）/ 範例模式（v2-v5 一鍵）
         ↓
[FastAPI：POST /api/v1/contracts/compare]
         ↓
[Parser + HTML Track-Change Stripper]  →  乾淨條款文字
         ↓
[Alignment]  →  條款對齊（LCS + Needleman-Wunsch）
         ↓
[Diff Engine]  →  新增 / 修改 / 刪除清單（含 old_text / new_text / change_type）
         ↓
[Risk Rule Engine]  →  risk_flag / level / trigger_reason
         ↓
[LLM Summary Service（Gemini / Claude fallback）]
  ├─ 白話摘要
  ├─ 重點收斂（3-5 個主要變更）
  └─ 基本協商方向
         ↓
[Report Generator  →  JSON + Markdown 報告]

── 按需呼叫 ──────────────────────────────────────
[POST /api/v1/contracts/negotiate]
  ↓
[Negotiate Service]
  ├─ Static Playbook（13 種風險類型，tier1 / tier2 / redline）
  └─ LLM 精煉（以 old_text + new_text 為脈絡，輸出替換條款文字）
```

**核心原則**：Rule Engine 做判斷（確定性，不受幻覺影響），LLM 做解釋（白話摘要與協商對策）。Phase 1.5 在 LLM 解釋層加入 MAS 雙 Agent 交叉驗證，進一步提升解讀精準度。

**MAS 設計（Phase 1.5）**：

```text
[高風險 flag]
  → async parallel
  ┌─────────────────────┐
  │ Agent A（嚴格）      │  極度保守的買方法律顧問
  │ Agent B（平衡）      │  促成交易的商務法務調解人
  └─────────────────────┘
  → Judge 決策矩陣
  confirmed（一致）→ ✓ 雙重驗證
  pending（分歧） → ⚠ 待確認 + 展示兩者觀點衝突
  single_agent   → 靜默降級（API 超時容錯）
```

### 驗證數字

| 指標 | 結果 | 說明 |
| --- | --- | --- |
| High-risk recall | **100%** | 高風險條款一筆不漏 |
| Overall detection | 61% | 保守設計，寧可高判不漏判 |
| 測試樣本 | 38 筆 gold set | v1 vs v2-v5 人工標註驗證 |

## 四、Demo 展示

### 啟動方式

```bash
# Terminal 1 — 啟動後端
uvicorn src.api.main:app --host 0.0.0.0 --port 8000

# Terminal 2 — 開啟 UI
open frontend/demo.html
```

### 建議 Demo 流程（競賽現場）

1. 開啟瀏覽器 `http://localhost:8000/demo`（後端需先啟動）
2. 點「範例模式」→ 選「**v4 保護刪減**」（最戲劇化，6 個高風險）
3. 展示 Loading 動畫（4 個階段橫向步驟）
4. 展示結果：Hero 指標條（26秒 / 100% / 雙軌）→ 整體評估 → 主要變更展開
5. 點開高風險條款 → 點「**📋 生成三層協商對策**」→ 展示 🟢🟡🔴 三層與替換條款文字
6. 展開「原文對照」→ 看 attribution 標籤（乙方修改／刪除）
7. 點「**下載 Markdown 報告**」

### 實際輸出（v1 vs v4，已驗證）

```text
# 合約比對報告

原始版本：SLA-like Base Contract v1.md
修訂版本：sla_v4_remove_protection.md
總變動：13 處（高風險 6、中風險 3、低風險 0）

## 整體評估
🔴 高度風險，建議協商後再簽署，必要時請法務審閱

## 主要變更重點

1. 🔴 保護條款刪除（第 4.4 條）
   觸發：保護性條款遭刪除
   對策：要求恢復、或以附件維持約束效力

2. 🔴 責任上限變更（第 11.1 條）
   觸發：賠償計算基礎從 12 個月縮至當月服務費
   對策：要求恢復原計算基礎，明列不適用上限情形

3. 🔴 廣泛免責擴大（第 11.2 條）
   觸發：排除間接損害、資料損毀、營業中斷、第三人求償
   對策：要求限縮免責範圍，明列例外情形

## 審閱建議

必須協商：第 4.4、8.4、11.1、11.2、11.3、12.3 條
建議協商：第 12.4、13.3、14.3 條
```

## 六、未來展望

### 短期（3 個月內）

- 支援 **NDA、採購合約**類型
- **合約範本 Playbook 管理**：企業上傳己方標準合約作為基準範本，系統自動以此為基準比對對方版本，標記所有偏差——範本邏輯反向設計，確保 AI 輸出符合公司既有商業立場，不憑空生成
- 整合 **Taiwan Law MCP**：風險分析自動引用相關法條（個資法、民法）
- 整合 **Microsoft Teams**：比對完成後自動通知責任人、SharePoint 自動歸檔

### 中期（6 個月內）

- **Insight 分析**：合約罰款條款變更時，自動查詢對方公司資本額，判斷罰款是否合理
- **歷史脈絡**：追蹤歷次合約修改記錄，提供談判趨勢分析
- 支援掃描版 PDF（OCR）

### 長期（12 個月）

- **週報整併**：自動彙整多份合約審查週報為月報
- **RBAC 權限分層**：不同職級看到不同敏感度資訊
- 推廣至其他部門（採購、業務、人資合約）

## 七、商機與推廣潛力

**內部**：法務、採購、業務三個部門均有合約審查需求，估計每週節省人工工時 10+ 小時

**外部商機**：

- 台灣中大型企業普遍缺乏法務資源，合約審查是痛點
- 可包裝為 SaaS 服務，針對 **科技業 IT 外包合約**、**零售業供應鏈合約** 切入
- 與 Lumine AI 定位互補，可作為其下游風險分析模組，未來整合進同一工作流

**市場規模估算**：以 Copilot 授權費為參考基準，企業版合約審查工具年費 $3,000-$10,000 / 席次，台灣目標市場 500 家企業，潛在規模約 $1.5B-$5B NTD

## 八、技術選型

| 元件 | 選擇 | 理由 |
| --- | --- | --- |
| LLM | Gemini 3.1 Flash Lite（主）/ Claude Sonnet 4.6（備） | 速度快、成本低；繁中效果佳 |
| 文件解析 | pdfplumber + python-docx + HTML stripper | 文字版 PDF/DOCX/MD，自動清理追蹤修改標記 |
| 後端 | FastAPI (Python 3.11+) | 輕量，與 Claude API 整合快 |
| 向量 DB | PostgreSQL + pgvector | 不增加新系統，條款語意檢索 |
| 認證 | Azure AD（現有） | 沿用公司既有系統 |
| 部署 | Azure VM | 合約資料留在企業內網 |
