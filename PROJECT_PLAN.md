# 合約智能比對助理 — 專案說明

**最後更新**: 2026-06-17  
**定位**: 讓法務與 PM 在 **30 分鐘內**完成過去需要 2 小時的合約審查  
**一句話**: 「像有一個資深法務顧問幫你審合約」  
**當前狀態**: 🟢 Demo 完整版完成（核心管道 + FastAPI + Demo UI）

---

## 一、解決什麼問題

每次供應商送來新版合約，法務或 PM 需要：

1. 人工逐條比對新舊版本（1-2 小時）
2. 判斷哪些變更有風險（需要法律知識）
3. 想出協商對策（需要談判經驗）

**現有工具的缺口**：Lumine AI 等工具只列出差異，不告訴你哪個重要、風險多高、怎麼談。

---

## 二、我們做了什麼

### 核心功能（已實作）

| 功能 | 說明 |
|------|------|
| **差異比對** | 逐條找出新增／修改／刪除條款，標記位置 |
| **智能摘要** | 100 個差異 → **3-5 個主要變更**，濾掉行政性小改動 |
| **風險分析** | 高／中／低三級標示，含商業影響說明 |
| **對策建議** | 每個高風險項目給出 2-3 個可直接用於協商的方案 |

### 與競品差異

| 功能 | 本系統 | Lumine AI |
| --- | --- | --- |
| 差異比對 | ✅ | ✅ |
| 重點摘要（3-5 點） | ✅ | ❌ |
| 風險等級標示 | ✅ | 部分 |
| **協商對策建議** | ✅ | ❌ |
| 繁體中文 SLA 優化 | ✅ | ❌ |

### 技術架構

```
[Demo UI：frontend/demo.html]
  上傳模式（drag & drop）/ 範例模式（v2-v5 一鍵）
         ↓
[FastAPI：POST /api/v1/contracts/compare]
         ↓
[Parser]  →  條款結構
         ↓
[Alignment]  →  條款對齊
         ↓
[Diff Engine]  →  新增 / 修改 / 刪除清單
         ↓
[Risk Rule Engine]  →  risk_flag / level / trigger_reason
         ↓
[LLM Summary Service]
  ├─ 白話摘要
  ├─ 重點收斂（3-5 個主要變更）
  └─ 協商對策（2-3 個可行方案）
         ↓
[Report Generator  →  JSON + Markdown 報告]
```

**核心原則**：Risk Rule Engine 做判斷與標記，LLM 做解釋與表達。LLM 不決定原始風險等級，只負責把已標記的風險翻成白話並給對策。詳見 [docs/architecture/service_design.md](docs/architecture/service_design.md)。

## 三、Demo 展示

### 輸入

- 合約原版（`sla_contract/SLA-like Base Contract v1.md`）
- 合約修訂版（`sla_contract/sla_v2_degrade.md` 至 `sla_v5_termination.md`）

### 執行方式

```bash
python3 -m src.services.contract.orchestrator \
  sla_contract/"SLA-like Base Contract v1.md" \
  sla_contract/sla_v4_remove_protection.md \
  --output samples/report_v1_vs_v4.md
```

### 實際輸出（v1 vs v4，已驗證）

```
# 合約比對報告

原始版本：SLA-like Base Contract v1.md
修訂版本：sla_v4_remove_protection.md
總變動：13 處（高風險 6、中風險 3、低風險 0）

## 整體評估
🔴 高度風險，建議協商後再簽署，必要時請法務審閱

## 主要變更重點

1. 🔴 保護條款刪除（第 4.4 條）
   觸發：保護性條款遭刪除
   對策：要求恢復、或以附件維持約束效力

2. 🔴 責任上限變更（第 11.1 條）
   觸發：賠償計算基礎從 12 個月縮至當月服務費
   對策：要求恢復原計算基礎，明列不適用上限情形

3. 🔴 廣泛免責擴大（第 11.2 條）
   觸發：排除間接損害、資料損毀、營業中斷、第三人求償
   對策：要求限縮免責範圍，明列例外情形

## 審閱建議

必須協商：第 4.4、8.4、11.1、11.2、11.3、12.3 條
建議協商：第 12.4、13.3、14.3 條
```

## 四、未來展望

本次 Demo 聚焦 SLA 合約核心流程。後續可擴展至：

### 短期（3 個月內）

- 支援 **NDA、採購合約**類型
- 整合 **Taiwan Law MCP**：風險分析自動引用相關法條（個資法、民法）
- 整合 **Microsoft Teams**：比對完成後自動通知責任人、SharePoint 自動歸檔

### 中期（6 個月內）

- **Insight 分析**：合約罰款條款變更時，自動查詢對方公司資本額，判斷罰款是否合理
- **歷史脈絡**：追蹤歷次合約修改記錄，提供談判趨勢分析
- 支援掃描版 PDF（OCR）

### 長期（12 個月）

- **週報整併**：自動彙整多份合約審查週報為月報
- **RBAC 權限分層**：不同職級看到不同敏感度資訊
- 推廣至其他部門（採購、業務、人資合約）

## 五、商機與推廣潛力

**內部**：法務、採購、業務三個部門均有合約審查需求，估計每週節省人工工時 10+ 小時

**外部商機**：

- 台灣中大型企業普遍缺乏法務資源，合約審查是痛點
- 可包裝為 SaaS 服務，針對 **科技業 IT 外包合約**、**零售業供應鏈合約** 切入
- 與 Lumine AI 等國際產品相比，本工具針對**台灣法律用語與商業慣例**優化，是本土差異化優勢

**評估**：以 Copilot 授權費為參考基準，企業版合約審查工具市場年費 $3,000-$10,000 / 席次，台灣目標市場 500 家企業，潛在規模約 $1.5B-$5B NTD

## 六、技術選型

| 元件 | 選擇 | 理由 |
| --- | --- | --- |
| LLM | Claude Sonnet 4.6 | 長文本理解強，繁中效果佳 |
| 文件解析 | pdfplumber + python-docx | 文字版 PDF/DOCX，穩定可靠 |
| 後端 | FastAPI (Python 3.11+) | 輕量，與 Claude API 整合快 |
| 向量 DB | PostgreSQL + pgvector | 不增加新系統，條款語意檢索 |
| 認證 | Azure AD（現有） | 沿用公司既有系統 |
| 部署 | Azure VM | 合約資料留在企業內網 |
