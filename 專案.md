## 1. 簡介

### 1.1 報告主題簡介

合約文件（SLA/NDA/採購）需人工比對差異，容易遺漏異動。

### 1.2 所選應用場景簡述

針對企業內部之合約（如 SLA、NDA、採購合約）進行版本管理與內容差異比對，聚焦 CLM（合約生命週期管理）的 **Review → Negotiation** 兩個核心環節。

### 1.3 報告目標與範圍

自動比對合約版本差異，快速標示異動內容，提供風險等級分析與協商對策，並可產出常用合約比對報告。

---

## 2. 應用場景描述

### 2.1 應用背景介紹

企業內部各單位（如零售服務部、營運發展部等）每年處理大量合約，現有合約具備個資與法律議題，部分合約（如 PIC 合約）仍以紙本形式保存。

國際研究指出：

- 法務人員有 **40–60%** 工作時間耗費在合約起草與審閱（Thomson Reuters）
- 手動合約流程平均週期 **3.4 週**（Agiled / DocuSign CLM Statistics）
- 專業法務人工審閱一份 NDA 平均耗時 **92 分鐘**（LawGeex vs 20 Lawyers Study）

台灣方面，工研院、勤業眾信台灣均確認本地企業面臨版本控管困難、法務審查瓶頸等相同痛點。

### 2.2 主要挑戰與痛點

合約文件（SLA/NDA/採購）在比對差異時高度仰賴人工，容易產生：

- 多版本並存，難以追蹤真正異動的條款
- 高風險條款（責任上限、免責範圍）遭修改但被忽略
- 知道有風險，但缺乏具體協商對策

### 2.3 痛點對現有流程的影響

低效合約審查流程平均使新產品上市推遲 **6.5 天**，對大型企業造成約 **700 萬美元**營收損失（Gartner Research）；合約管理不善使企業流失高達 **9.2%** 年營收（WorldCC / IACCM）。

## 3. 生成式 AI 解決方案

### 3.1 如何利用生成式 AI 解決上述痛點

本系統採用**雙軌架構**：

- **確定性規則引擎（Risk Rule Engine）**：硬性偵測高風險條款異動（責任上限縮減、免責範圍擴大、保護條款刪除），保證 **100% 召回率**，不依賴 LLM 判斷，避免 AI 幻覺造成漏判。
- **LLM（大型語言模型）**：負責白話摘要與協商對策生成。LLM 不決定風險等級，只對規則引擎已標記的風險進行解釋與建議，將「有哪些風險」轉化為「可直接使用的談判方案」。

### 3.2 解決方案概述及流程

**三段式提速設計**：

```text
AI 初審（26 秒完成差異比對與風險標記）
  → 人工確認（30 分鐘以內）
  → 附協商對策直接談判
```

**技術流程**：

```text
使用者上傳兩份合約（v1 / v2）
  → 文件解析（pdfplumber / python-docx）
  → 條款對齊（LCS + Needleman-Wunsch 演算法）
  → 差異偵測（新增 / 修改 / 刪除清單）
  → Risk Rule Engine（高/中/低風險標記）
  → LLM 摘要服務（3–5 個主要變更 + 協商對策）
  → 報告輸出（JSON + Markdown + Demo UI）
```

**對比 LawGeex 基準**：人工 NDA 審閱 92 分鐘，AI 初審 26 秒，AI 準確率 94% vs 人工 85%。

### 3.3 使用生成式 AI 的策略

- **文本比對**：LCS + Needleman-Wunsch 演算法進行條款級段落對齊，過濾標點排版干擾，只呈現實質異動。
- **風險解讀**：規則引擎偵測偏差類型後，LLM 根據已標記的風險生成白話說明與 2-3 個妥協方案。
- **成本控制**：不進行整份合約從頭掃描，只將差異段落與風險條款送入 LLM，大幅降低 Token 消耗。

## 4. 工具選擇與技術說明

### 4.1 選用的生成式 AI 工具

| 用途 | 工具 | 說明 |
| --- | --- | --- |
| 合約差異比對與摘要 | Claude Sonnet 4.6 | 長文本理解強，繁體中文效果佳 |
| 測試替代方案 | Gemini 3.1 Flash Lite | 成本低，適合快速測試與 Demo |
| 文件解析 | pdfplumber + python-docx | 支援文字版 PDF 與 DOCX |
| 後端 API | FastAPI（Python 3.11+） | 輕量，與 Claude / Gemini API 整合快 |

> ⚠️ **注意**：原文件提及 Gemini 3.5 Flash、GPT 5.4/5.5 為**尚未正式發佈的型號**，實際使用版本見上表。

### 4.2 工具選擇原因及優勢

- **Claude Sonnet 4.6**：對繁體中文法律條文理解準確，長合約（50+ 頁）不截斷。
- **雙軌架構（規則引擎 + LLM）**：規則引擎保證不漏判，LLM 做白話解釋，最符合企業法務安全需求——不能完全信任 AI，但也不能缺少 AI 的解釋能力。
- **FastAPI**：無需額外伺服器配置，本地即可執行，Demo 時穩定可靠。
- **pdfplumber / python-docx**：文字版 PDF 與 DOCX 解析穩定，不依賴 OCR，準確率高。

### 4.3 相關技術簡述

- **LCS（最長公共子序列）**：用於段落級比對，找出新舊版本間最相似的條款配對。
- **Needleman-Wunsch 演算法**：源自生物序列比對，用於條款對齊，處理插入／刪除條款的情境。
- **Risk Rule Engine**：基於正規表達式與關鍵詞模式的確定性規則系統，對應 CUAD 資料集 41 種核心合約條款類型（13,000+ 標注）。
- **LLM Prompt Design**：System Prompt 定義角色（資深法務顧問），User Prompt 提供已標記的差異條款與風險類型，引導 LLM 只做解釋不做判斷。

## 5. 實施細節

### 5.1 實施步驟與預期成果

1. 以標準合約範本（SLA / NDA / 採購）建立基準版 v1，再生成 v2–v5 差異版本作為驗證資料。
2. 實作文件解析層（PDF / DOCX → 條款結構）。
3. 實作條款對齊與差異引擎（LCS + Needleman-Wunsch）。
4. 建立 Risk Rule Engine，覆蓋責任上限、免責範圍、保護條款等高風險類型。
5. 串接 LLM API，生成白話摘要、重點收斂（3–5 個主要變更）與協商對策。
6. 建立 FastAPI 後端（`POST /api/v1/contracts/compare`）與 Demo UI。
7. 以 38 筆人工標註 gold set 驗證高風險召回率。

### 5.2 生成式 AI 技術的具體應用方法

不進行整份合約的從頭掃描，而是：

1. 規則引擎先確定哪些差異屬於高/中/低風險。
2. 只將已標記的風險條款送入 LLM，要求生成白話說明與協商方案。
3. LLM 輸出格式化的 JSON，前端直接渲染，不做二次解析。

### 5.3 整體架構圖

```text
[Demo UI：frontend/demo.html]
  上傳模式（拖曳上傳）/ 範例模式（v2–v5 一鍵）
         ↓
[FastAPI：POST /api/v1/contracts/compare]
         ↓
[Parser]  →  條款結構（PDF / DOCX）
         ↓
[Alignment]  →  條款對齊（LCS + Needleman-Wunsch）
         ↓
[Diff Engine]  →  新增 / 修改 / 刪除清單
         ↓
[Risk Rule Engine]  →  risk_flag / level / trigger_reason
         ↓
[LLM Summary Service]
  ├─ 白話摘要
  ├─ 重點收斂（3–5 個主要變更）
  └─ 協商對策（2–3 個可行方案）
         ↓
[Report Generator]  →  JSON + Markdown 報告
```

## 6. 預期效果與分析

### 6.1 預期改善效果

- 多版本差異比對從人工 1–2 小時縮短至 **AI 初審 26 秒**
- 高風險條款（責任上限、免責擴大、保護條款刪除）**100% 召回**，不因 AI 幻覺漏判
- 法務或 PM 在 **30 分鐘內**完成過去需要 2 小時的審查流程
- 每個高風險項目附帶 2–3 個可直接用於協商的方案，縮短談判準備時間

### 6.2 成果衡量方式

| 指標 | 結果 | 說明 |
| --- | --- | --- |
| High-risk recall | **100%** | 高風險條款一筆不漏 |
| Overall detection | 61% | 保守設計，寧可高判不漏判 |
| 測試樣本 | 38 筆 gold set | v1 vs v2–v5 人工標註驗證 |

**成本衡量**：單次 API 呼叫僅傳送已標記差異條款（非整份合約），每次分析 Token 消耗可控；正式上線後可依用量計費。

### 6.3 可能的風險或限制

- 合約具備個資與法律議題，應搭配存取控管（如 SharePoint / Azure Blob Storage）避免資安疑慮。
- 各家公司合約格式差異大，初期需針對特定類型（SLA / NDA）進行規則調校。
- 部分舊有合約為掃描版 PDF（圖片），目前系統僅支援文字版，OCR 支援列為後續項目。
- AI 輸出僅供輔助參考，最終決策須由法務人員確認。

## 7. 結論與未來展望

### 7.1 專題報告總結

本系統已完成核心管道實作，包含差異比對引擎、Risk Rule Engine、LLM 摘要服務、FastAPI 後端與 Demo UI，並以 38 筆人工標註 gold set 驗證，高風險召回率達 **100%**。現階段可完整展示 v1 vs v2–v5 的差異比對、風險標記與協商對策生成流程。

### 7.2 未來建議

- 實務導入時應「規範合約格式」或「限定適用場景」（先從 SLA 開始），讓系統在明確範圍內提供更穩定的比對結果。
- 建議與法務部門合作建立企業專屬的 **Playbook 條款庫**，以公司既有立場為基準自動標記偏差。

### 7.3 未來優化或拓展

#### 短期

- 支援 NDA、採購合約類型
- 建立企業專屬 **Playbook 條款庫**，以公司既有立場為基準自動標記偏差

#### 中期：M365 企業整合

目前系統以 FastAPI + 本地規則引擎為核心，後續可整合 M365 原生資源，降低導入門檻：

- **SharePoint**：合約上傳後自動觸發比對流程，確保資料留存於企業內網，符合個資與法律要求
- **Power Automate**：合約進入 SharePoint 時自動分類、重點擷取，再呼叫比對 API，減少人工操作
- **Teams Adaptive Cards**：比對完成後直接在 Teams 頻道推送風險摘要與協商對策，取代 Email 通知，保留審核軌跡
- **Lumia AI（Lumi API）**：可作為 LLM 替代方案接入，企業若已有 Lumia 授權可無縫切換

此架構可無縫延伸至公司內部其他有導入 O365 的 BU，且掌握低 Token 成本的解決方案（規則引擎前置過濾），將成為商用化的競爭優勢。

#### 長期

- 整合台灣法條 MCP（民法、個資法）；追蹤歷次合約修改趨勢
- RBAC 權限分層；推廣至採購、業務、人資合約等其他部門
