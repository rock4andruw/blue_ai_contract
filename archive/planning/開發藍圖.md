## 核心產品藍圖

建議系統拆成 6 個核心模組，這樣既能 demo，也能逐步產品化。
這些模組與主流 AI 合約平台的能力相近，尤其 clause extraction、risk flagging、redlining、playbook comparison 與可追溯審閱，是目前市場可商業化的基本盤。**monday**+2

## 1. 文件匯入模組

* 支援 DOCX、PDF 上傳。
* 建立文件 metadata：合約類型、版本、部門、客戶、日期。
* 先做格式正規化，讓後續條款比對穩定。**justee**+1

## 2. 條款解析模組

* 自動切分 clause / section。
* 識別條款標題與編號。
* 建立 clause-level 結構資料，供 diff 與檢索使用。**litera**+1

## 3. 差異比對模組

* 比對 A 版與 B 版合約。
* 標示新增、刪除、修改、位移。
* 補上「修改摘要」，不只是純文字差異。**monday**+1

## 4. 標準條款 / Playbook 比對模組

* 對照公司條款庫與 fallback 規則。
* 標示「偏離標準」「需法務覆核」「高風險」。
* 可支援部門別或客戶別審閱規則。**sirion**+1

## 5. 範本生成模組

* 根據合約類型與輸入欄位產生初稿。
* 優先採用模板組裝，不直接讓 LLM 全文自由生成。
* 對使用者來說，這是最容易感受到效率提升的功能。**hyperstart**+1

## 6. 審閱工作台模組

* 顯示原文、異動、風險、建議替代條文。
* 允許人工接受 / 拒絕 AI 建議。
* 保留審計軌跡與版本紀錄，這對商業落地很重要。**goheather**+2

## 技術架構

你的產品應採 **規則 + LLM + 知識庫 + 人工覆核** 的混合架構，而不是單純一個聊天模型。
現在主流 legal AI 平台強調的也是結構化抽取、條款知識、風險規則與 workflow automation，而不是讓模型直接自由發揮。**bindlegal**+2

## 建議四層架構

| 層級   | 內容                            | 說明                                 |
| ------ | ------------------------------- | ------------------------------------ |
| 接入層 | Web UI / 上傳 / 權限            | 給使用者上傳合約、查看結果           |
| 處理層 | Parser / Diff / Extraction      | 做條款切分、差異偵測、欄位抽取       |
| 智慧層 | LLM + Clause Library + Playbook | 做風險判斷、摘要、替代建議、範本生成 |
| 治理層 | Review Log / Audit / Metrics    | 做版本控管、人工確認、可觀測性       |

## 技術設計原則

* **規則先行** ：條款編號、標題、版本 diff 先靠 deterministic 邏輯做。[docusign](https://www.docusign.com/blog/best-ai-legal-contract-analysis-tools)
* **LLM 補強** ：用於語意摘要、風險說明、替代條文生成。**bindlegal**+1
* **RAG 檢索** ：檢索標準條款、歷史範本、公司政策。**hyperstart**+1
* **Human-in-the-loop** ：最終審閱決策由人做，不做全自動法律結論。**hsfkramer**+2

## 競賽版 MVP

內部競賽最重要的是： **做出 1 個畫面就能理解價值的 MVP** 。
我建議 MVP 只做一個場景： **兩版 NDA 合約差異比對 + 標準條款偏離標示 + 一鍵輸出報告** ，因為這最能短時間展示商業感。**docusign**+1

## MVP 功能清單

* 上傳舊版與新版 NDA。
* 自動切分條款。
* 顯示 clause-level 差異。
* 標示是否偏離公司標準 NDA。
* 生成簡短審閱摘要。
* 匯出報告（PDF / DOCX / HTML）。**monday**+1

## MVP 不要做的事

* 不要一開始支援太多合約類型。
* 不要先做完整多 agent。
* 不要先做複雜簽核流。
* 不要把資源花在漂亮但無法證明價值的功能上。

## 成功指標

如果你的目標是比賽拿名次又能賣，評估就不能只看模型準確率。
CLM 與合約管理最佳實務都建議用 cycle time、compliance、value realization、risk reduction 這類 KPI 來衡量成果。**hyperstart**+1

## 技術 KPI

* 條款切分正確率。
* 差異偵測正確率。
* 風險標示 precision / recall。
* 產出報告時間。
* 系統平均回應時間。

## 商業 KPI

* 合約審閱首輪時間縮短比例。**sirion**+1
* 法務人工比對工時下降比例。[sirion](https://www.sirion.ai/library/contract-insights/ai-playbook-redlining-vs-manual-contract-review/)
* 非標準條款漏看率下降。
* 範本重用率上升。
* 使用者願意持續使用的意願。

## 競賽 KPI

* 5 分鐘 demo 能否看懂價值。
* 是否能展示真實 before / after。
* 是否能提出清楚的 target customer。
* 是否能說明怎麼賣、賣給誰、為何會付錢。

## 風險控管

這種專題最常失敗的原因，不是技術做不出來，而是做出來卻不能信。
現在 legal AI 市場普遍要求 explainability、audit trails、human oversight 與治理機制，所以你的產品一定要把「AI 只是輔助，結果可回溯」講清楚。**hsfkramer**+2

## 必備風險控制

* 顯示 AI 判斷依據來自哪一條原文。[goheather](https://goheather.io/post/the-9-best-ai-contract-review-tools-for-2026)
* 高風險建議需人工確認。[lexcheck](https://blog.lexcheck.com/human-in-the-loop-vs.-fully-automated-ai-contract-review-software-lc)
* 每次接受 / 拒絕 AI 建議都留紀錄。[contractcorridor](https://contractcorridor.com/contractsexplained/9-contract-management-best-practices-for-2026-clm-tips-strategies/)
* 區分一般使用者與法務管理者權限。
* 不宣稱提供最終法律意見，而是提供審閱輔助。[hsfkramer](https://www.hsfkramer.com/insights/2025-12/2026-the-year-ai-and-legal-technology-became-business-as-usual)
