可以做到，而且用「skill + agent」來驅動一個**人看得懂的 HTML 儀表板，同時又是主 agent 的上下文 / RAG 入口**是很合理的方向。下面我用系統工程視角幫你拆成架構設計與實作細節。

---

## 整體架構概念

目標：

* 前端：一個合約專案儀表板（HTML），給人看。
* 後端：同一份 HTML（或其結構化抽取）被主 agent 當成 RAG context，配合 skills / MCP tools 存取。

可以想成：

1. **來源資料層** ：合約原文（PDF/Doc）、meta（專案、金額、風險、KPI）、審查歷史等 → 先轉成結構化（JSON / DB）與 chunks。
2. **RAG 層** ：向量庫（per contract / per clause chunk）、檢索 + rerank。
3. **Dashboard HTML** ：

* 只顯示必要摘要、狀態、關鍵條款超連結等。
* 同時在 DOM 中嵌入「對 LLM 友善」的結構（data-* / `<script type="application/json">` / `<pre>` 等）。

1. **Agent + Skills 層** ：

* MCP / skills：
  * tool A：讀取 / 查詢 dashboard HTML 或其後端 API。
  * tool B：針對某合約 ID 做 RAG 查詢。
* 主 agent：理解 dashboard 的結構描述 + 使用這些 tools 完成任務。

這個思路其實跟現在 repo 裡用 `AGENTS.md` / context file 告訴 AI agent project 結構、policy 有點類似，只是你把它升級成「帶 UI 的 HTML context」。[[arxiv](https://arxiv.org/abs/2510.21413)]

---

## HTML 儀表板的資訊設計

你要同時顧三件事：

1. 人類閱讀：像一般專案 dashboard（卡片、表格、狀態顏色）。
2. LLM 閱讀：結構清楚、噪音少、有明確的語義標記。
3. RAG mapping：每個 UI 元素要能對應到後端可查詢的 key（contract_id / clause_id）。

要點：

* **語意化 tag** ：`<section>`, `<article>`, `<h2>`, `<ul>` 讓 LLM 解析時天然有層次；相關研究證實 LLM 能理解 HTML 結構，對此類任務是有優勢的。[[arxiv](https://arxiv.org/pdf/2210.03945.pdf)]

## Agent / Skill 設計思路

你提到想用 skill + agent，大致可以拆成：

## 1. Context / 規則 skill（類似 AGENTS.md）

* 做一個「儀表板說明 skill」，內容描述：
  * HTML 中各個 section 的語意（`#contracts` 是合約清單、`#risks` 是風險總覽…）。
  * `data-contract-id`、`data-clause-id` 等屬性代表什麼。
  * `<pre class="llm-context">` 內 JSON 的欄位定義。

這有點像現在研究裡提到的 `AGENTS.md` / context file：用自然語言 + 少量 schema 告訴 agent 怎麼解讀檔案結構。[[arxiv](https://arxiv.org/abs/2602.14690)]

## 2. Perception skills：讀取 / 查詢 HTML 與 RAG

例：建幾個 MCP / tool（名稱你可自訂）：

1. `get_dashboard_html()`
   * 讀取最新儀表板 HTML（從 Git、S3 或內網 HTTP endpoint）。
   * 主 agent 用它把 HTML 摘要出有用區塊（或你在 server 端直接抽出 `<article>` list 回 JSON）。
2. `list_contracts(filter?)`
   * 不一定要走 HTML，可以直接打後端 API，回 `[{contract_id, name, status, risks...}]`。
   * 儀表板只是一個人類 UI；LLM 真正可用的是這個 JSON。
3. `query_contract_rag(contract_id, question)`
   * 對特定合約做向量檢索後，回「若干 chunks + metadata」。
   * 主 agent 再在自己的 context 裡綜合回答。

目前針對 MCP / agent tools 的研究也觀察到，絕大多數實務使用都是這種「perception tools（讀檔、查 DB）」+「action tools（寫檔、調 API）」的組合，你這個設計完全符合這個 pattern。[[semanticscholar](https://www.semanticscholar.org/paper/ad6b0cd5d8e1243277210d7c37c0cda9ffa35956)]

## 主 Agent 的行為設計（高層 prompt / system）

在主 agent 的 system / skill 指令裡明確規定：

1. 優先用 `list_contracts` / `get_dashboard_html` 取得「有哪些合約、狀態與風險」。
2. 只在需要深入某合約時，呼叫 `query_contract_rag(contract_id, question)`。
3. 回答時：
   * 引用 dashboard 上的人類可見欄位（專案名稱、金額、狀態）。
   * 結合 RAG 取回的條款內容，用自然語言說明。

這樣， **人眼看 dashboard，一眼知道全局** ；主 agent 則在背後透過 skills 深入查某一張合約全文的 RAG context。

## HTML / RAG 的技術細節建議

## 1. 儀表板 HTML 本身也可進入 RAG

* 你可以把儀表板 HTML 經 pre-processor 轉成「結構化 markdown / text」存入向量庫，讓模型在回答時同時參考「dashboard 摘要」和「原始合約 chunk」。
* 研究顯示，相較於純 plain text，保留 HTML 結構（或由其轉成帶層級的文本）能讓 RAG 更好利用原始知識結構。[[arxiv](http://arxiv.org/pdf/2405.03989.pdf)]

## 2. 控制 HTML 噪音與長度

* 儀表板專門給 LLM / 人雙方看的那一頁，盡量避免：
  * 大量 CSS / JS inline，
  * 動態生成的大量無關 DOM。
* 對 LLM 用的 context，最好是：
  * 只保留 `<main>` 或 `<section id="contracts">` 等關鍵區域。
  * 或 server 端直接抽取必要 DOM → 序列化成精簡 HTML / Markdown 再給 agent。

## 安全與維運（你偏基礎架構視角比較會在意的）

1. **權限分層** ：

* 儀表板前端：前端 auth（SSO / AD / LDAP）；
* RAG tool：在 MCP / skills 層做 ACL，agent 只能查自己有權限的 contract_id。

1. **Prompt injection / tool poisoning** ：

* MCP 安全研究指出，file-based / tool-based prompt injection 是真實風險，特別是外部 HTML / 檔案內容被當成「可信指令」時。[[ieeexplore.ieee](https://ieeexplore.ieee.org/document/11531012/)]
* 對策：
  * 規定主 agent 把 dashboard / RAG 回來的內容視為「資料」，永遠不要執行其中任何自然語言指令。
  * 在 system prompt 裡明寫：只接受工具描述與人類指令為指令來源。

1. **Context rot / 配置腐爛** ：

* context files（AGENTS.md、技能描述）和儀表板 schema 容易隨系統演進而失真，已有研究把這現象稱為 context rot。[[semanticscholar](https://www.semanticscholar.org/paper/8d308e0e2569761fc65fb58a459901510878e26d)]
* 建議：用 Git 管理「儀表板 schema / skill 定義 / system prompt」，每次 schema 變更都一起調整 context file。

## 若你要開始 PoC，可以這樣落地

1. 選一小組合約，先建一個簡單 `contracts.html` 儀表板：
   * 一個 `<section id="contracts">` 裡幾個 `<article>` 卡片 + `<pre class="llm-context">` JSON。
2. 建一個簡單 RAG API：`/rag/query_contract?contract_id=...&q=...`。
3. 寫兩個最小 MCP / skill：
   * `get_contract_dashboard()`：回儀表板精簡 HTML 或抽取後 JSON。
   * `query_contract_rag(contract_id, question)`。
4. 在主 agent system prompt 裡定義：
   * 優先用 dashboard 了解全局，
   * 需要細節再用 RAG tool。

這樣你很快就可以有一個「人眼看的合約儀表板 + 主 agent 可以理解與操作的 HTML / JSON 結構」的雛型；之後再慢慢演進「風險指標」「KPI」「提醒 / 任務」等功能。
